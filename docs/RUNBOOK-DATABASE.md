Operations guide for PostgreSQL database management on Railway and Docker deployments.

**Database:** PostgreSQL 16
**ORM:** SQLAlchemy 2.x (async) + Alembic
**Hosting:** Railway (production), Docker Compose (staging/self-hosted)

## 1. Backups

### Automatic Backups (Docker Compose)

The `backup` service in `docker-compose.prod.yml` runs backups automatically:

- **Schedule:** Twice daily at 03:00 and 15:00 UTC
- **Retention:** 7 days local (configurable via `BACKUP_RETAIN_DAYS`)
- **Format:** `alfredo_<db>_<timestamp>.sql.gz`
- **Location:** `backup_data` Docker volume (mount at `/backups`)

### Manual Backup (Any Environment)

```bash
# Docker Compose
docker compose -f docker-compose.prod.yml exec backup /usr/local/bin/backup-db.sh

# Direct (local psql available)
PGHOST=localhost PGUSER=postgres PGPASSWORD=xxx PGDATABASE=saas_platform \
  ./scripts/backup-db.sh

# Railway — use the Railway CLI to connect, then dump
railway run pg_dump --no-owner --no-acl --clean --if-exists | gzip -9 > backup_$(date +%Y%m%d).sql.gz
```

### Copy Backup Out of Docker Volume

```bash
# Find the backup container
docker compose -f docker-compose.prod.yml exec backup ls -la /backups/

# Copy to host
docker compose -f docker-compose.prod.yml cp backup:/backups/alfredo_saas_platform_20260320_030000.sql.gz ./
```

### Offsite Backup (S3/R2)

Set these environment variables to enable automatic S3 upload:

```env
BACKUP_S3_BUCKET=my-bucket-name
AWS_ACCESS_KEY_ID=xxx
AWS_SECRET_ACCESS_KEY=xxx
```

The backup script will upload to `s3://<bucket>/backups/<filename>` after each successful local backup.

## 2. Restore

### From Backup File

```bash
# Interactive (will ask for confirmation)
./scripts/restore-db.sh /path/to/alfredo_saas_platform_20260320.sql.gz

# Non-interactive (CI/CD or scripted)
./scripts/restore-db.sh /path/to/backup.sql.gz --force
```

**What it does:**
1. Terminates active connections
2. Drops and recreates the database
3. Restores from the backup file
4. Re-grants permissions to the `saas_app` user
5. Verifies table count and key data

### Restore in Docker Compose

```bash
# 1. Stop the backend to prevent writes
docker compose -f docker-compose.prod.yml stop backend

# 2. Execute restore inside the backup container
docker compose -f docker-compose.prod.yml exec backup \
  /bin/sh -c 'PGHOST=db PGUSER=$PGUSER PGPASSWORD=$PGPASSWORD PGDATABASE=$PGDATABASE \
  restore-db.sh /backups/alfredo_saas_platform_20260320_030000.sql.gz --force'

# 3. Restart everything
docker compose -f docker-compose.prod.yml up -d
```

### Restore on Railway

```bash
# 1. Download backup to local machine
# 2. Get the Railway Postgres connection string from the dashboard
# 3. Restore directly
gunzip -c backup.sql.gz | psql "postgresql://user:pass@host:port/dbname" --single-transaction
```

## 3. Migrations

### Run Pending Migrations

```bash
# Docker Compose
docker compose -f docker-compose.prod.yml exec backend alembic upgrade head

# Local development
cd backend && alembic upgrade head

# Railway
railway run alembic upgrade head
```

### Stamp Existing Database (First Time Only)

If the database already has all tables but Alembic has never been run:

```bash
alembic stamp 001_initial
```

This tells Alembic that the DB is at the `001_initial` revision without running any DDL.

### Create a New Migration

```bash
# Auto-generate from model changes
cd backend
alembic revision --autogenerate -m "add_new_field_to_unidades"

# Review the generated file in alembic/versions/
# ALWAYS review autogenerated migrations before applying
```

### Check Current Revision

```bash
alembic current   # Shows current DB revision
alembic history   # Shows migration history
alembic heads     # Shows latest revision
```

### Rollback a Migration

```bash
# Roll back one step
alembic downgrade -1

# Roll back to a specific revision
alembic downgrade 001_initial
```

## 4. Monitoring

### Check Backup Health

```bash
# Docker Compose
docker compose -f docker-compose.prod.yml exec backup /usr/local/bin/check-backup-health.sh

# Direct
BACKUP_DIR=/path/to/backups ./scripts/check-backup-health.sh
```

Returns exit code 0 if healthy, 1 if not. Checks:
- Backup directory exists
- At least one backup file present
- Latest backup is less than 48 hours old
- File size is reasonable (> 1KB)
- Gzip integrity is valid

### Monitor Active Connections

```sql
-- Current connections by user
SELECT usename, count(*) FROM pg_stat_activity
WHERE datname = 'saas_platform' GROUP BY usename;

-- Long-running queries (> 30 seconds)
SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '30 seconds'
  AND state != 'idle';
```

### Monitor Table Sizes

```sql
SELECT relname AS table, pg_size_pretty(pg_total_relation_size(relid)) AS size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;
```

### Monitor Index Usage

```sql
SELECT relname, idx_scan, seq_scan,
       CASE WHEN idx_scan + seq_scan > 0
            THEN round(100.0 * idx_scan / (idx_scan + seq_scan), 1)
            ELSE 0
       END AS idx_usage_pct
FROM pg_stat_user_tables
ORDER BY seq_scan DESC LIMIT 20;
```

## 5. Emergency Procedures

### Kill a Runaway Query

```sql
-- Find the query
SELECT pid, now() - pg_stat_activity.query_start AS duration, query
FROM pg_stat_activity
WHERE state = 'active' AND query NOT LIKE '%pg_stat_activity%';

-- Cancel it (graceful)
SELECT pg_cancel_backend(<pid>);

-- Terminate it (forceful)
SELECT pg_terminate_backend(<pid>);
```

### Database Is Out of Connections

```sql
-- Check current connections
SELECT count(*) FROM pg_stat_activity WHERE datname = 'saas_platform';

-- Kill idle connections
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE datname = 'saas_platform'
  AND state = 'idle'
  AND query_start < now() - interval '10 minutes';
```

If persistent, increase connection limit:

```sql
ALTER DATABASE saas_platform CONNECTION LIMIT 100;
```

### Table Is Locked / Deadlock

```sql
-- Find blocking queries
SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.usename AS blocked_user,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query AS blocked_statement,
       blocking_activity.query AS current_statement_in_blocking_process
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
  AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
  AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
  AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
  AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
  AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
  AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
  AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
  AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
  AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

-- Kill the blocking process
SELECT pg_terminate_backend(<blocking_pid>);
```

### Disk Space Running Low

```sql
-- Find largest tables
SELECT schemaname, relname, pg_size_pretty(pg_total_relation_size(relid))
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC;

-- Run VACUUM to reclaim space
VACUUM (VERBOSE, ANALYZE);

-- For a specific bloated table
VACUUM FULL VERBOSE actividades;
```

### Corrupt Index

```sql
-- Reindex a specific table
REINDEX TABLE CONCURRENTLY unidades;

-- Reindex everything (use with caution)
REINDEX DATABASE CONCURRENTLY saas_platform;
```

## 6. Railway-Specific

### Access the Database

```bash
# Via Railway CLI (opens a proxy)
railway connect postgres

# Get connection string
railway variables --service postgres
```

### Railway Backups

Railway provides automatic daily backups on Pro plans. To take a manual backup:

```bash
# Use Railway CLI to proxy the connection
railway run pg_dump --no-owner --no-acl --clean --if-exists \
  | gzip -9 > alfredo_railway_$(date +%Y%m%d_%H%M%S).sql.gz
```

### Railway Restore

```bash
# 1. Get the DATABASE_URL from Railway dashboard
# 2. Restore directly (Railway databases accept standard psql connections)
gunzip -c backup.sql.gz | psql "$DATABASE_URL" --single-transaction
```

### Scaling on Railway

- **CPU/Memory:** Adjust in Railway service settings
- **Connection pooling:** Use PgBouncer if connections exceed 50
- **Read replicas:** Not natively supported; use Railway volumes or external managed Postgres

### Running Alembic on Railway

```bash
# Option A: Railway CLI
railway run alembic upgrade head

# Option B: Add to your Dockerfile entrypoint
# In backend/Dockerfile:
#   CMD ["sh", "-c", "alembic upgrade head && uvicorn app.main:app ..."]
```

### Hardening Railway Database

```bash
# Run hardening script via Railway proxy
railway run ./scripts/harden-db.sh
```
